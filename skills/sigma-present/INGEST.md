# Ingest: artifact → output mapping

How `sigma-present` turns each sigma markdown artifact into deck slides or report
sections. All artifacts live under `sigma/specs/{date}-{slug}/`.

## General rules

- **H1 (`#`)** → deck title slide / report hero `<h1>`.
- **H2 (`##`)** → one slide (DECK) or one `<section class="reveal-on-scroll">` (REPORT).
- **H3 (`###`)** → sub-heading within the slide/section.
- **Bullets** → `<li>`; in DECK mode, wrap as reveal fragments for incremental
  reveal when the list is an argument being built up.
- **Code fences** → kept verbatim; DECK uses reveal highlight plugin.
- **Tables** → HTML `<table>` (both modes).
- **Citations / links** → collected into a Sources section/end-slide.
- **Fact vs inference** → sigma marks inferred claims; render inference text with
  the `.inference` class (italic, muted) and factual/cited claims normally.

## DECK conversion mechanism

reveal.js parses markdown directly. Feed the artifact body into the deck
template's `<textarea data-template>`:
- Slide breaks: insert a line containing only `---` between H2 sections (the
  template's `data-separator` is `^\n---\n$`).
- Vertical (drill-down) slides: separate with a line containing only `--`.
- Speaker notes: append `Note:` blocks, or fill `{{SPEAKER_NOTES}}`.

## REPORT conversion mechanism

Render each H2 block to semantic HTML and wrap in
`<section class="reveal-on-scroll">…</section>`, concatenated into `{{SECTIONS}}`.
Hero `{{TITLE}}`/`{{LEDE}}` come from the H1 and the first paragraph.

## Per-artifact mapping

| Artifact | Default mode | Mapping |
|----------|-------------|---------|
| `research.md` | REPORT (or DECK) | H1 → hero; H2 → sections; findings → callouts; **citations → footnotes/Sources**; fact vs inference styled distinctly; per-source confidence shown as a tag |
| `proposals.md` | DECK | each proposal → a slide; pros/cons → two-column or fragment list |
| `architecture.md` | DECK or REPORT | each component → slide/section; diagrams/code preserved; decisions as callouts |
| `spec.md` | REPORT | requirements → numbered sections; acceptance criteria → checklists |
| `tasks.md` | REPORT | phases → numbered sections; tasks → checklist cards; **Chart.js bar of tasks-per-domain** |
| plan (design doc) | REPORT | phases → sections; risk table → HTML table; build order → ordered list |
| verify output | REPORT | PASS/FAIL summary hero + **Chart.js doughnut (pass rate)**; per-check cards color-coded by verdict; logic-evaluator findings as a separate block |
| `board.md` / events | KANBAN | columns → flex/grid columns; cards → tickets with id+domain; **Chart.js doughnut of column counts**; status drives card border color |

## Charts

Use Chart.js (CDN, or inlined for offline). Pass the theme `--accent` palette
into dataset colors. Typical charts:
- tasks per domain → bar
- verify pass/fail → doughnut
- kanban column counts → doughnut (To Do / In Progress / Blocked / Done)

## Provenance

Always fill `{{STAGE}}`, `{{DATE}}`, `{{SLUG}}` from the artifact's workspace so
the footer reads: `Generated by sigma · {stage} · {date} · source: {slug}`.
